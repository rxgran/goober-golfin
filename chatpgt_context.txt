This is a very doable project in principle, but the hard part isn’t the app—it’s getting reliable ball flight data from a single phone camera in uncontrolled conditions. If you design it correctly from the start, though, you can absolutely build something useful for swing analysis + rough distance estimation.

Here’s a realistic breakdown of how to approach it.

1. First decide what you’re actually predicting

You’ll have way more success if you don’t try to fully recreate a TrackMan-level simulator.

Split the problem into two layers:

A. Swing analysis (easy → very doable)

From video alone:

swing tempo
club path (approx)
face angle proxy
swing speed estimate (rough)
consistency metrics

This is computer vision + pose estimation, very achievable on phone.

B. Ball flight + distance (hard part)

From a single camera:

launch direction (moderate)
launch angle (hard but possible)
carry distance (very hard without depth or multi-angle)

You will likely need:

either multiple camera angles
or strong assumptions + calibration
or a hybrid ML + physics model
2. Data strategy (your simulator idea is actually the key)

What you described is essentially building your own labeled dataset:

Inputs (X)

From phone video:

raw frames or extracted features:
pose keypoints
club position tracking
ball detection (pre/post impact frames)
optical flow near impact
Labels (Y)

From simulator / range system:

carry distance
total distance
launch angle
shot direction
ball speed (if available)
spin (if available, even better)

This becomes a supervised learning problem:

video → shot metrics

3. Computer vision pipeline (what you should actually build)
Step 1: Detect golfer pose

Use:

MediaPipe Pose (fast on mobile)
or MoveNet (more robust)

This gives:

shoulder rotation
hip rotation
wrist position
swing timing
Step 2: Detect club + ball (harder)

Options:

train a small object detector (YOLOv8n / YOLOv9-tiny)
or use motion-based detection (optical flow + frame differencing)

Key moment:

impact frame detection is critical
Step 3: Extract swing features

From pose + club tracking:

club head speed estimate (pixel velocity → calibrated scale)
swing plane angle
attack angle proxy
tempo (backswing vs downswing timing)
Step 4: Ball tracking (optional but powerful)

If visible:

detect first 3–10 frames after impact
fit trajectory curve

Even short tracking helps estimate:

launch direction
initial velocity vector
4. Modeling approach (start simple)

Don’t start with deep learning end-to-end. Start with:

Phase 1: Feature → regression model

Use:

XGBoost / Random Forest / simple MLP

Input:

swing features + early ball motion features

Output:

distance, launch angle, etc.

This gets you something working quickly.

Phase 2: sequence model (better)

Once you have more data:

LSTM or Temporal CNN
or lightweight Transformer over frames/features
Phase 3: end-to-end vision model (advanced)

Only after lots of data:

video → metrics directly
5. Key engineering problem: calibration

Without calibration, distance prediction will drift badly.

You’ll need:

known reference object (golf ball size helps)
fixed phone placement (tripod strongly recommended)
consistent camera angle
measured ground reference markers if possible

Even a cheap yard setup benefits massively from:

1–2 cones at known distances
6. MVP you should actually build first

If you try everything at once, it’ll fail. Start with:

MVP v1:
record swing video
detect pose
classify swing quality metrics:
tempo
balance
swing plane consistency

NO distance prediction yet.

MVP v2:
detect impact frame
estimate launch direction
rough distance estimate using learned model
MVP v3:
integrate simulator dataset training
personalize model per user (you)
7. Big reality check (important)

What you’re trying to replace (simulators like TrackMan / GCQuad) use:

radar + high-speed cameras + multi-sensor fusion

A phone alone:

struggles with depth
struggles with fast ball tracking (golf ball is tiny + fast)

So your success target should be:

“good enough feedback for backyard practice”
not perfect yardage accuracy.

8. Tech stack suggestion
Mobile app:
Flutter or React Native (if cross-platform)
or native Swift (best for camera performance)
CV:
MediaPipe (pose)
OpenCV (motion + tracking)
YOLOv8n (custom ball/club detection)
Backend (optional):
Python + FastAPI
PyTorch for training

Your best advantage is this:

You control the data collection environment.

That’s huge.

package com.example.goobergolfin

import android.Manifest
import android.os.Bundle
import androidx.activity.ComponentActivity
import androidx.activity.compose.setContent
import androidx.activity.enableEdgeToEdge
import androidx.activity.result.contract.ActivityResultContracts
import androidx.camera.core.CameraSelector
import androidx.camera.core.Preview
import androidx.camera.lifecycle.ProcessCameraProvider
import androidx.camera.view.PreviewView
import androidx.compose.foundation.layout.fillMaxSize
import androidx.compose.material3.Scaffold
import androidx.compose.runtime.Composable
import androidx.compose.ui.Modifier
import androidx.compose.ui.platform.LocalContext
import androidx.compose.ui.viewinterop.AndroidView
import androidx.core.content.ContextCompat
import androidx.lifecycle.compose.LocalLifecycleOwner
import com.example.goobergolfin.ui.theme.GooberGolfinTheme

class MainActivity : ComponentActivity() {

    private val requestPermission =
        registerForActivityResult(ActivityResultContracts.RequestPermission()) { }

    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)

        // Ask for camera permission
        requestPermission.launch(Manifest.permission.CAMERA)

        enableEdgeToEdge()

        setContent {
            GooberGolfinTheme {
                Scaffold(modifier = Modifier.fillMaxSize()) {
                    CameraPreview()
                }
            }
        }
    }
}

@Composable
fun CameraPreview() {
    val context = LocalContext.current
    val lifecycleOwner = LocalLifecycleOwner.current

    AndroidView(
        modifier = Modifier.fillMaxSize(),
        factory = { ctx ->
            val previewView = PreviewView(ctx)

            val cameraProviderFuture = ProcessCameraProvider.getInstance(ctx)

            cameraProviderFuture.addListener({

                val cameraProvider = cameraProviderFuture.get()

                val preview = Preview.Builder().build().also {
                    it.setSurfaceProvider(previewView.surfaceProvider)
                }

                val cameraSelector = CameraSelector.DEFAULT_BACK_CAMERA

                try {
                    cameraProvider.unbindAll()

                    cameraProvider.bindToLifecycle(
                        lifecycleOwner,
                        cameraSelector,
                        preview
                    )

                } catch (e: Exception) {
                    e.printStackTrace()
                }

            }, ContextCompat.getMainExecutor(ctx))

            previewView
        }
    )
}
